# Arquitetura (referência rápida)
Este documento existe para evitar regressões causadas por alterações sem contexto (tipos duplicados, services recriando entidades que já existem no schema, selects com colunas inexistentes, etc.). A ideia é que qualquer pessoa (ou IA) consiga entender o fluxo principal sem ler o projeto inteiro.

---

## 1) Visão geral da arquitetura

### Camadas e responsabilidades
Fluxo obrigatório de dados:

```
Page/Componente (UI)
      ↓ usa
Composables (useQuery / useMutation / usePagination / useFilter)
      ↓ chama
Services (1 arquivo por domínio)
      ↓ usa
lib/api.ts (wrapper de acesso ao Supabase)
      ↓ usa
lib/supabase.ts (cliente Supabase)
      ↓
Supabase (PostgREST/RPC) → PostgreSQL
```

Papéis:
- Page/Componente: renderização, interação do usuário, composição de hooks, tratamento de UX (loading, erro, toasts).
- Composables: estado e ciclo de vida de leitura/escrita (loading/erro/data), paginação e filtros. Não contém query SQL/REST.
- Services: lógica de dados por domínio (monta filtros, colunas, chama select/insert/update/rpc). Centraliza regras de negócio relacionadas a leitura/escrita.
- lib/api.ts: padroniza o acesso ao Supabase (select, insert, updateWhere, upsert, delete, rpc) e normaliza erros.
- lib/supabase.ts: instancia o client do Supabase.

### Fonte de verdade dos tipos
A fonte única de verdade para tipos de banco é o arquivo gerado:
- src/integrations/supabase/types.ts

Ele expõe helpers que devem ser usados no app:
- Tables<'nome_da_tabela'> (Row)
- TablesInsert<'nome_da_tabela'> (Insert)
- TablesUpdate<'nome_da_tabela'> (Update)
- Enums<'nome_do_enum'>

O app também re-exporta aliases amigáveis em:
- src/types/index.ts

Regra: services podem criar aliases para Tables<> (ex: `type PacientRow = Tables<'pacient'>`), mas não devem declarar interfaces/entidades “na mão” duplicando schema.

## 2) Fluxo de uma requisição (exemplo real: listar pacientes)

Arquivos envolvidos no fluxo real de listagem:

```
src/pages/Pacientes.tsx
  └── src/composables/useQuery.ts
        └── src/services/pacientes.service.ts
              └── src/lib/api.ts
                    └── src/lib/supabase.ts
```

Passo a passo:
1. A página dispara uma query via composable `useQuery`, passando uma função que chama o service.
2. O service monta `columns` + filtros (`eq/ilike/range/limit`) e chama `api.select(...)`.
3. `api.select` cria o builder do Supabase (`supabase.from(table).select(...)`), aplica filtros e retorna `{ data, error, count }` no formato padrão.
4. O `useQuery` gerencia `loading/error/data` e devolve para a página renderizar.

## 4) Regras e convenções (para IAs e para o time)

### Regras de tipagem (fonte de verdade)
- Sempre derive tipos de entidade do schema gerado via `Tables<>/TablesInsert<>/TablesUpdate<>` em [types.ts](file:///c:/Users/caual/Documents/Projetos%20Pessoais/Nuvia/nuvia-orchestrator/src/integrations/supabase/types.ts#L680-L757).
- Em services: permitido criar aliases (ex.: `type PacientRow = Tables<'pacient'>`), proibido recriar interfaces de entidades manualmente.
- Em selects: nunca peça coluna que não exista no `Row` daquela tabela.

### Regras de acesso a dados (camadas)
- Página/componente não fala com o banco: deve chamar composables, que chamam services.
- Services não devem importar `supabase` diretamente; devem usar `api.ts`.
- Exceção: [auth.service.ts](file:///c:/Users/caual/Documents/Projetos%20Pessoais/Nuvia/nuvia-orchestrator/src/services/auth.service.ts) usa `supabase.auth` por ser um módulo separado de autenticação.

### Regras de segurança/isolamento multi-org
- Sempre filtrar por `organization_id` em toda query que retorne dados de tabelas multi-tenant.
- Nunca “confiar” em UI para isolar dados; o filtro precisa existir no service.

### Regras de relacionamento (joins) no Supabase
- Relações no `select` devem usar a sintaxe de join do Supabase (PostgREST), e não “coluna escalar que não existe”.
  - Exemplo de padrão (apenas ilustrativo): `role:role_id(role_id,name)`

### Soft delete
- Não remover registros via delete físico quando o domínio usa soft delete.
- Padrão do projeto: usar `active: false` (e filtrar `active: true` nas listagens).

### UX/erro obrigatório
- Sempre tratar o `error` retornado pelos composables (ex.: toast, alert).
- Evitar busca manual com `useState + useEffect` para dados remotos; usar `useQuery`.

## 5) Exemplo completo anotado (código real)

### 5.1 Página chamando o service via composable
Trecho real de [Pacientes.tsx](file:///c:/Users/caual/Documents/Projetos%20Pessoais/Nuvia/nuvia-orchestrator/src/pages/Pacientes.tsx#L45-L76):

```tsx
const pagination = usePagination({ pageSize: 10 })
const { page, range, setTotal } = pagination
const { filters, setFilter, debouncedFilters, clearFilters, isDirty } = useFilter({
  initial: { search: '' },
  debounceMs: 400,
})

const listQuery = useQuery(
  () =>
    pacientesService.listar({
      search: debouncedFilters.search,
      range,
      count: 'exact',
    }),
  {
    immediate: true,
    deps: [debouncedFilters.search, page],
    onError: (err) => toast.error(err.message),
  }
)

useEffect(() => {
  if (typeof listQuery.count === 'number') {
    setTotal(listQuery.count)
  }
}, [listQuery.count, setTotal])
```

O que está acontecendo aqui:
1. A página não conhece Supabase nem SQL: ela só chama `pacientesService.listar(...)`.
2. `useFilter` faz debounce do texto para evitar múltiplas chamadas a cada tecla.
3. `usePagination` fornece `range` (from/to) para o service usar no `api.select`.
4. `useQuery` padroniza `loading/error/data` e expõe `refetch`.

### 5.2 Service montando filtros/colunas e chamando a API
Trecho real de [pacientes.service.ts](file:///c:/Users/caual/Documents/Projetos%20Pessoais/Nuvia/nuvia-orchestrator/src/services/pacientes.service.ts#L28-L51):

```ts
const columns =
  'pacient_id, organization_id, name, phone, email, birth_date, status, notes, created_at, active'

export const pacientesService = {
  async listar(input: ListPacientsInput): Promise<ApiResponse<PacientRow[]>> {
    const { search, range, limit, count } = input

    const opts: SelectOptions = {
      columns,
      eq: {
        active: true,
      },
      order: [{ column: 'created_at', ascending: false }],
      range,
      limit,
      count,
    }

    if (search && search.trim()) {
      opts.ilike = { name: `%${search.trim()}%` }
    }

    return api.select<PacientRow[]>(TABLE, opts)
  },
}
```

O que este service garante:
1. `columns` concentra as colunas permitidas/esperadas naquele endpoint.
2. Os filtros são montados em um objeto `SelectOptions` (que é entendido por `api.select`).
3. A tipagem do retorno usa `Tables<'pacient'>` (alias `PacientRow`) vindo do schema gerado.

Observação importante:
- Em tabelas multi-tenant, `organization_id` deve entrar no `eq` do service para evitar vazamento de dados entre organizações.

### 5.3 API wrapper criando a query e normalizando retorno
Trecho real de [api.ts](file:///c:/Users/caual/Documents/Projetos%20Pessoais/Nuvia/nuvia-orchestrator/src/lib/api.ts#L145-L183):

```ts
async select<T = unknown>(
  table: string,
  opts: SelectOptions = {}
): Promise<ApiResponse<T>> {
  try {
    let query: any = supabase
      .from(table)
      .select(opts.columns ?? '*', {
        count: opts.count ?? undefined,
      })

    query = applyFilters(query, opts)

    if (opts.order) {
      for (const o of opts.order) {
        query = query.order(o.column, { ascending: o.ascending ?? true })
      }
    }

    if (opts.range) {
      query = query.range(opts.range.from, opts.range.to)
    } else if (opts.limit) {
      query = query.limit(opts.limit)
    }

    if (opts.single) {
      query = query.single()
    }

    const { data, error, count } = await query

    if (error) return { data: null, error: normalizeError(error) }

    return { data: data as T, error: null, count: count ?? undefined }
  } catch (err) {
    return { data: null, error: normalizeError(err) }
  }
}
```

O que a camada `api` padroniza:
1. Sempre retorna `{ data, error, count? }` no mesmo formato.
2. Concentra a lógica de filtros/ordenação/paginação em um só lugar (`applyFilters` + `.order/.range/.limit`).
3. Centraliza a normalização de erros (`normalizeError`) para a UI não depender do formato do Supabase.
